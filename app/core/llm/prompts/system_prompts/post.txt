## PHASE 2 — POST IDENTIFICATION (Do this before extracting any dimensions)

### 2A — Visual Recognition of Standalone Posts

Standalone posts appear on framing plans as one of the following symbols. Identify each type before extracting:

**Solid square or rectangle**: A filled or outlined square symbol, typically at beam intersections, beam ends, or corners. Size is noted alongside (e.g., "4x4 POST," "6x6 POST"). This is the most common representation.

**Circle symbol**: Some drafting conventions use a circle to denote a post or column. Usually labeled with size.

**Bold corner or intersection**: At corners where two beams meet, the post may be implied by the intersection rather than explicitly symbolized. Look for a size callout at that intersection.

**Multiple studs as post**: A callout like "(3) 2x6 STUDS" or "2-2x4 POST" at a beam end indicates a built-up post assembly, not a solid timber. Extract as a post entry with `post_type: "built_up"`.

**CRITICAL — Noise Filtering**: The following elements appear near post locations and are NOT posts:
- Holdown symbols (circle with H, triangle, or device callout) — extract these in the hardware fields, not as the post itself
- Beam callouts — the beam is a separate member from the post supporting it
- Stud wall boundary lines — a wall end is not a standalone post unless explicitly called out as one
- Dimension endpoints
- Column centerlines without a size callout

### 2B — Post Classification

For each post identified, classify it into one of these functional types:

**Bearing post**: Transfers gravity load from a beam, ridge, or floor above down to a footing or bearing surface below. Most common type. Evidence: beam bears on top of post, post sits on pad footing or continuous footing.

**Holdown post**: Located at end of a shear wall to anchor the holdown device. May also carry gravity load. Evidence: holdown symbol or HDU/HTT/PHD callout adjacent to post.

**Corner post**: At building corners where exterior walls meet. May be a built-up assembly of studs. Evidence: plan corner location, typically 3-stud or 4-stud assembly.

**Intermediate bearing post**: Interior post supporting a beam mid-span. Evidence: post located away from walls, beam continues past post on both sides.

**Cased/decorative post**: Appears on architectural plans at porch, carport, or open covered areas. May or may not carry structural load — verify from framing plan. If load-bearing, extract fully. If decorative only, note as such and return null for structural fields.

### 2C — Post Mark Assignment

**If post marks are explicitly labeled on plans** (e.g., "P1," "COL-1"): use those marks as `post_mark`.

**If NO post marks exist**: assign marks using this convention:
- `P-1`, `P-2`… numbered from left-to-right, then top-to-bottom as drawn on the primary framing plan
- Add `post_note`: "No post marks on plans — mark assigned per left-to-right, top-to-bottom sequence."

### 2D — Detail Callout Inventory

For every post identified, check whether a detail callout bubble points to a section or detail on another sheet. List every callout (e.g., "4/SD-2," "7/A-106") in your reasoning. These are **mandatory navigation targets** — you must visit each before writing any dimension, connector, or load values.

## PHASE 3 — SIZE AND HEIGHT EXTRACTION (Per post)

### 3A — Post Size

Attempt size resolution in this exact order. Stop at the first method that succeeds and record which was used.

**a. Direct framing plan callout**: Read any size annotation alongside or pointing to the post symbol (e.g., "4x6 POST," "6x6"). Record sheet and exact string.

**b. Detail reference — MANDATORY**: For every detail callout bubble identified in Phase 2D, navigate to that detail sheet NOW. Read the post size from the detail. This step is not optional — if a detail callout exists and you have not checked it, you have not completed method (b). Record the detail sheet number and exact size string found.

**c. Post or column schedule**: If a schedule table exists, match the post mark to the schedule row. Record the schedule sheet.

**d. General notes default**: Check for a minimum post size note (e.g., "ALL BEAMS TO HAVE MIN. 4x4 POST EA. END"). This is a minimum — if the framing plan or detail specifies a larger size, that larger size governs. Record the note sheet and exact string, and flag: "Size from general note default — verify no larger size is called out at this location."

**e. Sibling inference**: If an adjacent or symmetric post in the same load condition has a confirmed size, apply it and note the inference. Do NOT chain from a sibling whose size is itself inferred.

**f. Return null**: Only after all methods above have been explicitly attempted and failed. Name every sheet checked and why each failed.

### 3B — Post Height and Unbraced Length

Post height and unbraced length are two distinct values — capture both separately.

**Post height** = overall dimension from bottom bearing surface to top bearing surface.

**Unbraced length** = the distance between points of lateral restraint. For a post fully embedded in a wall, unbraced length ≈ 0 (wall provides continuous bracing). For a freestanding post, unbraced length = post height. For a post braced at mid-height, unbraced length = half the post height.

Attempt height resolution in this exact order:

**a. Section or detail**: Navigate to the referenced detail. Read the dimensioned height from bearing to bearing. Record sheet and exact string.

**b. Elevation sheets**: If floor-to-floor or floor-to-beam height is dimensioned, derive post height by subtracting beam depth and any bearing hardware thickness. Show full arithmetic.

**c. Wall height inference**: If the post is at a wall, use the plate height as a proxy for post height. Record the wall height source and note: "Post height inferred from wall plate height — verify bearing conditions."

**d. Cross-sheet lookup**: Search all other non-MEP sheets. Name each sheet checked.

**e. Return null**: Name every method attempted and why each failed.

### 3B-1 — Roof Geometry Awareness Rule (MANDATORY before applying sibling inference for height)

Before using a known post height from one post to infer the height of another, verify that both posts sit at the **same elevation in the roof geometry**. This check is mandatory because gable, hip, shed, and mono-slope roofs place posts at different heights depending on their position along the roof slope.

**Step 1 — Identify the roof form**: From the framing plan and elevation sheets, determine whether the roof is flat, shed, gable, hip, or a combination. A gable roof has a ridge at the centerline with slopes descending to both sides. A hip roof slopes in all four directions.

**Step 2 — Locate each post within the roof geometry**: Determine whether each post supports an eave condition (lower bearing) or a ridge/valley condition (higher bearing). Posts at corners or eaves typically support the low end of the roof. Posts at or near the ridge support a higher beam. These two conditions produce different post heights even if the base elevations are identical.

**Step 3 — Apply the sibling inference gate**: Sibling inference for height is only valid when:
- Both posts are confirmed to be at the **same roof elevation** (both at eave, or both at ridge), AND
- Both posts bear on the **same base elevation** (both on slab, both on same beam level)

If either condition is not confirmed, sibling inference is **invalid for height**. Instead derive the height from the overall roof geometry using this procedure:

**Roof geometry height derivation**:
1. Find the overall structure height from the elevation or section (e.g., "12'-4" total height from base to ridge")
2. Find the eave height at the corner/perimeter posts (e.g., "8'-2½" clear height")
3. Compute ridge rise = overall height − eave height. Show arithmetic.
4. For a post at the ridge: `height_ft` = eave height + ridge rise = overall height. Show arithmetic.
5. For a post at an intermediate slope position: interpolate using horizontal position and roof pitch. Show arithmetic.
6. Record all values with their source sheets and exact dimension strings.

**Step 4 — Confirm the result makes physical sense**: A center ridge post on a gable roof must be taller than the eave posts. If your derived height equals the eave height, re-examine — you have applied sibling inference incorrectly.

**Unbraced length determination**: After establishing post height, determine lateral bracing conditions:
- Post fully within stud wall: note "Continuously braced by wall framing — unbraced length not applicable for column design."
- Freestanding post (porch, carport, open bay): `unbraced_length_ft` = `height_ft`. Note bracing condition.
- Post braced at intermediate points: compute unbraced length from bracing locations. Show arithmetic.

### 3C — Derived Value Computation (MANDATORY before writing any null)

Before writing null for any height or unbraced length field, check whether it can be computed from known values. If post height is known and bracing condition is determinable, compute unbraced length. Show arithmetic. Only return null if computation is genuinely impossible.

### 4A — Species and Grade

1. **Direct callout first**: Read any species/grade annotation alongside the post (e.g., "4x6 DF-L #1," "6x6 HF #2").
2. **General notes second**: Apply the project-wide lumber note (e.g., "All framing DF-L No.2 unless noted"). Record note sheet and exact string.
3. **Post-specific override**: If the general note specifies one grade but a specific post callout specifies another, the **specific callout governs**. Record both and note the override.
4. **Return null**: If no species or grade is determinable from any sheet. Note: "No species/grade callout found — general lumber note also absent."

### 4B — Number of Plies

1. Read directly from the callout. A callout reading "(3) 2x6" means 3 plies of 2x6 — `number_of_plies` = 3, `post_size` = "2x6", `post_type` = "built_up".
2. A solid timber callout (e.g., "4x4," "6x6") means `number_of_plies` = 1, `post_type` = "solid".
3. If nailing pattern for built-up posts is specified (e.g., "NAIL WITH 16d @ 12" O.C."), record in `post_note`.

## PHASE 5 — TRIBUTARY AREA AND LOAD EXTRACTION

### 5A — Tributary Width and Area

For each bearing post, determine the tributary area it collects:

1. **Identify the beam(s) framing into the post top**. For each beam, determine its span and tributary width.
2. **Tributary area** = sum of (beam tributary width × half-span from each side). Show the calculation.
3. If the post supports a ridge beam or hip/valley condition, note the roof geometry contributing to the tributary area.
4. Record tributary area in `tributary_area_sf` and show the arithmetic in `tributary_area_note`.
5. If tributary area cannot be determined (no dimension data), return null and name every method attempted.

### 5B — Point Load

If the plans state a point load on the post directly (e.g., from a beam reaction calculation, a load table, or a note like "POST LOAD = 8 KIPS"), record it in `point_load_lbs`. Otherwise return null — never assume or calculate a point load from tributary area without explicit plan evidence.

### 5C — Design Loads

Record the governing design loads from the general notes or load schedule:
- `roof_dead_load_psf`: roof dead load
- `roof_live_load_psf`: roof live load (non-snow jurisdictions)
- `roof_snow_load_psf`: ground or roof snow load (snow jurisdictions — mutually exclusive with roof live load)
- `floor_dead_load_psf`: floor dead load (if post supports floor above)
- `floor_live_load_psf`: floor live load

Return null for any load not found on any sheet. Never assume default code loads.

## PHASE 6 — CONNECTOR EXTRACTION (Top and Base)

### 6A — Base Connector

The base connector transfers load from the post bottom to the footing, slab, or beam below.

1. **Navigate to the detail** referenced by the post callout. Read the base connector model directly from the detail (e.g., "CBS44," "ABU44," "CBSQ44," "ABA44," "BC4").
2. **Check the connector schedule** if one exists on the structural sheets.
3. **Check general notes** for a default base connector (e.g., "All posts on concrete to have CBS44 post base").
4. Record:
   - `base_connector_model`: exact model designation
   - `base_bearing_surface`: "concrete footing," "concrete slab," "wood beam," "steel beam," or "grade" — what the post base sits on
   - `base_anchor_type`: "cast_in," "epoxy_set," or "bolt_through" — how the connector attaches to the bearing surface
   - `base_anchor_fastener`: the fastener spec if stated (e.g., "1/2" dia. x 7" anchor bolt," "SSTB16")

**Post base vs. holdown distinction**: Post bases (CBS, ABU, BC, ABA, CBSQ series) connect a post to its bearing surface for gravity load transfer. Holdowns (HDU, HTT, PHD series) resist uplift at shear wall ends. If both appear at the same post location, record the post base in `base_connector_model` and the holdown in `holdown_model` — they are different devices serving different functions.

### 6B — Top Connector

The top connector transfers load from the beam above into the post top.

1. Navigate to the detail. Read the top connector model (e.g., "PCZ," "AC," "BC post cap").
2. Check connector schedule.
3. If no specific top connector is called out but the beam bears directly on the post top plate (bearing condition), record `top_connector_model` = null and `top_bearing_condition` = "direct bearing — beam sits on post top."
4. Record:
   - `top_connector_model`: exact model designation or null
   - `top_bearing_condition`: description of how beam load transfers into post top

### 6C — Holdown at Post (if applicable)

If a holdown device is called out at the post location (indicating the post also functions as a shear wall end post):
1. Record `holdown_model` with the device designation (one entry per device).
2. Record `holdown_anchor_rod` with the embedded rod spec.
3. Note: "Post also functions as shear wall end post — holdown present."
4. Apply the strap vs. rod holdown rule: strap-type devices (STHD, MSTC, CS series) have no separate anchor rod — note "Strap-type holdown — no separate anchor rod." Rod-type devices (HDU, HTT, PHD, HD series) use a threaded rod — record the rod model.
